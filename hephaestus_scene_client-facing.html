All external services and SDKs this project connects to at runtime or during the build pipeline. None of these are optional — removing any one would require code changes.

Unity Gaming Services
UNITY CCD

Cloud Content Delivery — remote CDN for VPT scene bundles

Hosts the compressed Addressable asset bundles for all 9 VPT factory rooms (01_TourEntrance through 09_WarehouseAndDistribution). On each app launch the headset fetches a remote catalog, compares content hashes, and downloads only changed bundles — no App Lab submission needed for content updates. Bundles are cached on-device so no network connection is required during play.

UNITY AUTHENTICATION

Anonymous sign-in — silent session token for Cloud Code access

Runs automatically when the player enters 00_WaitingRoom. No login screen is shown — the SDK signs the player in anonymously and returns a session token. That token is required to authorise any calls to Unity Cloud Code. Without it, the access code gate cannot function.

UNITY CLOUD CODE

Server-side access code validation — CheckAccessCode module

The player types a code on the in-VR keypad in the WaitingRoom. That code is sent server-side to the VPTAccessService/CheckAccessCode Cloud Code module, which returns true or false. No validation logic lives on the device — codes can be managed without rebuilding the app.

UNITY BUILD AUTOMATION

CI/CD pipeline — automated Addressables build and CCD upload

CloudBuildConfigurator.cs reads the CLOUD_PROFILE_NAME environment variable set in Unity Build Automation to determine which CCD bucket to target. When a build is triggered, Addressables bundles are automatically packaged and uploaded to CCD as part of the pipeline — no manual upload step required.

Third-party services
MIXPANEL

Analytics — training event tracking across KTA and VPT

Tracks the following events: Environment_Entered, Environment_Exited, Level_Started, Level_Completed, Level_Exited, Module_Entered, Module_Exited, plus timed event durations. All tracking is handled by AnalyticsManager.cs. Data is sent to the Mixpanel project associated with the app's API key — no on-device storage of analytics data.

Hardware SDK
META XR SDK

Quest controller haptics and in-editor VR simulation

Two Meta XR packages are included: com.meta.xr.sdk.haptics (67.0.0) for controller vibration feedback during interactions, and com.meta.xr.simulator (81.0.0) for testing the experience inside the Unity editor without a physical headset. Neither package communicates with an external server at runtime — both are local device SDKs.

The items below are open questions surfaced from a technical review of the Unity project. Each one represents either a known gap, an assumption baked into the codebase that needs client confirmation, or a decision that will affect future maintenance and updates.

Critical — needs resolution before launch
⚠ ORPHAN SCENE

01_StartingRoom is not in the Addressable/CCD system. Was this intentional, or should it be migrated?

Every other VPT room (02–09) is hosted on Unity CCD and can be updated without an App Lab submission. 01_StartingRoom is missing from the Addressable group entirely — the code falls back to SceneManager.LoadSceneAsync, loading it directly from the APK. This means any change to that scene requires a full binary rebuild and store review. The script StartingRoomAditivePortal.cs is explicitly marked TODO / temp in the codebase. If this is a permanent scene, it should be added to the Addressable group so it can be updated remotely like its siblings.

Content & access management
ACCESS CODES

What is the intended lifecycle of VPT access codes — single-use, time-limited, or permanent?

The Cloud Code module CheckAccessCode validates a typed code server-side and returns true/false. No expiry or usage-count logic is visible in the project bindings. If codes are meant to be single-use (e.g. issued per training session) or time-limited (e.g. valid for 30 days), that logic needs to be built into the Cloud Code module. Who currently manages issuing and revoking codes, and through what interface?

KTA ACCESS

Is the KTA showroom intentionally open to anyone with the app installed, or should it also have an access gate?

The access code gate only exists in the VPT flow (00_WaitingRoom). The KTA showroom (Lobby → Commercial / Residential) has no equivalent authentication step — any headset with the app can explore it freely. If the KTA content is meant to be demo-only or partner-facing, a similar gate should be added. If it's intentionally open, no action is needed.

CCD DEPLOYMENT

Is there a staging vs. production CCD environment, and who approves content pushes to production?

The CloudBuildConfigurator.cs script reads a CLOUD_PROFILE_NAME environment variable set in Unity Build Automation to determine which CCD bucket to target. It's not clear from the project whether a staging bucket exists for testing content updates before they go live. A bad push to the production bucket would affect every headset on next launch. Who owns the CCD project credentials and what is the release approval process?

Analytics & reporting
ANALYTICS SCOPE

Are the current Mixpanel events sufficient, or are there additional training KPIs the client needs?

Currently tracked: Environment_Entered, Environment_Exited, Level_Started, Level_Completed, Level_Exited, Module_Entered, Module_Exited, plus timed events. Common training KPIs that are not currently tracked include: access code failure rate, time spent per factory room, completion rates by headset model or firmware version, and whether players are using the Facility Map shortcut vs. walking linearly through portals. Should any of these be added?

ANONYMOUS AUTH

Is anonymous sign-in the intended long-term authentication model, or will users eventually have named accounts?

Unity Authentication is currently used in anonymous mode — a silent session token is created on each launch, purely to authorise Cloud Code calls. There is no persistent player identity across sessions or headsets. If the client later wants per-user training records, completion certificates, or leaderboards, the auth model would need to be upgraded to named accounts. Is anonymous auth the intended long-term approach?
